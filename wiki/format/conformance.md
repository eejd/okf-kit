# Overview

OKF v0.1 is intentionally permissive (SPEC §9). Conformance is a three-way classification — errors, warnings, info — and only **errors** block conformance. The philosophy: bundles stay useful as they grow, get refactored, and are partially generated by agents, so a consumer must tolerate missing optional fields, unknown types, extension keys, and broken links without falling over.

# Definition

A bundle is conformant **iff** it has no errors. The error classes are:

- **frontmatter-missing** — a non-reserved `.md` with no `---` block on line 1.
- **frontmatter-invalid** — a present-but-broken block (non-mapping YAML, or unparseable).
- **type-empty** — frontmatter present but `type` is missing or blank (the only required field).
- a present-but-invalid block on a reserved `index.md` / `log.md`.

Warnings (non-blocking) cover missing recommended `title` / `description`, concept ids that violate the segment regex, and broken internal links. Info (non-blocking) covers extension frontmatter keys, a nested `index.md` carrying frontmatter (a forward-compat sub-bundle marker — see [Reserved files](/format/reserved-files.md)), `okf_version` state, and empty bundles.

# Examples

The parser never raises on malformed input; it records a degraded concept and lets the validator decide. So conformance lives in one place — `validate_bundle` — and both the CLI and MCP call it (see [validate module](/core/validate.md)). The CLI exits 1 on errors, 0 otherwise; the MCP `validate` tool returns the same `{conformant, errors, warnings, info}`. The permissive stance is documented in [Permissive parsing](/architecture/permissive-parsing.md).
